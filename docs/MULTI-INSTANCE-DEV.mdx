# Multi-Instance Development (Shared Infra)

Sentris Flow supports running multiple isolated dev instances (0-9) on one machine.

## Quick Start

```bash
# Instance 0 (default) — works exactly as before
just dev
bun run dev

# Check active instance for this workspace
just instance show
bun run instance show

# Persist instance 1 for this workspace
just instance use 1
bun run instance use 1
just dev          # now starts instance 1
bun run dev       # also starts instance 1

# Stop your instance
just dev stop
bun run dev:stop
```

## How It Works

- **One shared Docker infra stack**: Postgres, Temporal, Redpanda, Redis, MinIO, etc.
- **Many app instances** via PM2: `sentris-{backend,worker,frontend}-N`
- **Isolation via namespacing**, not per-instance containers:
  - Postgres database: `sentris_instance_N`
  - Temporal namespace + task queue: `sentris-dev-N`
  - Kafka client/group IDs: `sentris-*-N`

## Selecting an Instance

The instance is resolved in this order:

1. `SENTRIS_INSTANCE` environment variable (highest priority)
2. `.sentris-instance` file in repo root (gitignored)
3. Initialize `.sentris-instance` to `0` and use instance `0`

```bash
# Per-command
SENTRIS_INSTANCE=2 just dev
SENTRIS_INSTANCE=2 bun run dev

# Per-workspace (persistent)
just instance use 2
bun run instance use 2
just dev
bun run dev
```

If `just` is not available, use `bun run instance show` and `bun run instance use N` for the same workspace-scoped selector. `bun run instance show` creates `.sentris-instance` with `0` when no active instance has been selected, so later scripts have an explicit target. The value must be an integer from `0` to `9`; malformed values fail fast before ports, PM2 app names, or databases are computed. The cross-platform `bun run dev [start|stop|logs|status|clean]` and `bun run dev:stop` scripts use the same resolution order as the rest of local tooling, and `bun run dev` initializes or repairs `.instances/instance-N/{backend,worker,frontend}.env` before PM2 starts.

## Instance Env Files

PM2 can read per-instance env files from:

- `.instances/instance-N/backend.env`
- `.instances/instance-N/worker.env`
- `.instances/instance-N/frontend.env`

Use `just instance-env` to manage them:

```bash
# Omit N to use SENTRIS_INSTANCE, then .sentris-instance, initializing it to 0 if missing
just instance-env show

# Create env files for an instance (uses app/.env, or falls back to app/.env.example)
just instance-env init 2

# Re-apply only instance-scoped values (ports, DB, Temporal namespace)
just instance-env update 2

# Copy env files between instances and rescope values
just instance-env copy 2 5

# Show file presence + computed values
just instance-env show 5
```

`just dev` auto-initializes env files for the active instance when missing (without overwriting existing files).

## Port Map

Ports are offset by `N * 100`:

| Service  | Base | Instance 0 | Instance 1 | Instance 2 | Instance 5 |
| -------- | ---- | ---------- | ---------- | ---------- | ---------- |
| Frontend | 5173 | 5173       | 5273       | 5373       | 5673       |
| Backend  | 3211 | 3211       | 3311       | 3411       | 3711       |

Shared infra (fixed ports, same for all instances):

| Service          | Port        |
| ---------------- | ----------- |
| Postgres         | 5433        |
| Temporal         | 7233        |
| Temporal UI      | 8081        |
| Redis            | 6379        |
| Redpanda (Kafka) | 9092        |
| Redpanda Console | 8082        |
| MinIO API/UI     | 9000 / 9001 |
| Loki             | 3100        |

## Nginx Limitation

The nginx reverse proxy (`http://localhost`) always routes to **instance 0** (ports 5173/3211 are hardcoded in `docker/nginx/nginx.dev.conf`). This is by design — nginx is shared infra.

For non-zero instances, access your app directly:

```
# Instance 1
http://localhost:5273        # frontend
http://localhost:3311/api    # backend API
```

The Vite dev server proxies `/api` calls to the correct backend port automatically (computed from `SENTRIS_INSTANCE` in `vite.config.ts`).

## Commands

All commands respect `SENTRIS_INSTANCE` (or the active instance selected by `just instance use N` / `bun run instance use N`):

```bash
# Select instance once per workspace
just instance use 1
bun run instance use 1

# Start
just dev

# Stop (only stops PM2 apps; infra stays running for other instances)
just dev stop
bun run dev:stop

# Logs (filtered to your instance's PM2 apps)
just dev logs

# Status
just dev status
bun run dev status

# Clean (stops PM2 apps; only tears down infra if instance 0)
just dev clean

# Reset the active instance database, then rerun migrations
just db-reset

# One-off override without changing active workspace instance
SENTRIS_INSTANCE=3 just dev
SENTRIS_INSTANCE=3 bun run dev
SENTRIS_INSTANCE=3 just db-reset
```

`just dev status` and `bun run dev status` show PM2 app state, shared Docker infrastructure state, and live runtime probes for the selected instance's frontend, backend liveness, backend readiness, and worker health endpoints. This catches the case where PM2 is already `online` but the app has not opened its HTTP port yet.

`just dev`, `bun run dev`, `just dev clean`, and `bun run dev clean` prune oversized PM2 logs for the selected instance before or after PM2 lifecycle actions. The default cap is 64MB per app log file (`sentris-{frontend,backend,worker}-N-{out,error}.log`). Override it for a local session with `SENTRIS_PM2_LOG_MAX_BYTES`.

The backend PM2 app watches `backend/src`, `backend/scripts/seed-templates`, and `packages/shared/src` so template catalog and validation metadata changes are reflected in the running API. The worker PM2 app watches its own source plus shared runtime package sources used during execution.

When stopping/cleaning instance 0, Docker infra is also torn down. For non-zero instances, only the PM2 apps are stopped (since other instances may still need the shared infra).

## Local Maintenance Scripts

Local maintenance scripts use the shared script runtime at `scripts/lib/local-script-runtime.ts`.

Scripts that mutate or inspect local Postgres data are instance-aware. They resolve their target database in this order:

1. Script-specific override, such as `TEMPLATE_SEED_DATABASE_URL`, `DELETE_WORKFLOW_RUNS_DATABASE_URL`, `MIGRATION_SMOKE_DATABASE_URL`, or `SEED_STRESS_DATABASE_URL`
2. Generic script override: `SENTRIS_SCRIPT_DATABASE_URL`
3. `SENTRIS_INSTANCE`
4. `.sentris-instance`
5. Initialize `.sentris-instance` to `0` and use instance `0`

These scripts intentionally do **not** use plain `DATABASE_URL` by default. `DATABASE_URL` belongs to the running app process, production Drizzle runs, and app env files; using it for local maintenance scripts can accidentally target the legacy `sentris` database instead of `sentris_instance_N`.

Local Drizzle migrations are covered by the same rule. `bun --cwd backend run migration:push` targets the active instance by default, ignores stale `backend/.env` `DATABASE_URL` values, and initializes `.sentris-instance` to `0` if needed. Use `DRIZZLE_DATABASE_URL` only when intentionally migrating a different database. In production (`NODE_ENV=production` or `SENTRIS_ENV=production`), Drizzle uses `DATABASE_URL`.

Scripts that start or inspect Temporal workflows resolve namespace and task queue in this order:

1. Script-specific override pair, such as `WORKFLOW_RUNNER_TEMPORAL_NAMESPACE` + `WORKFLOW_RUNNER_TEMPORAL_TASK_QUEUE`
2. Generic override pair: `SENTRIS_SCRIPT_TEMPORAL_NAMESPACE` + `SENTRIS_SCRIPT_TEMPORAL_TASK_QUEUE`
3. `SENTRIS_INSTANCE`
4. `.sentris-instance`
5. Initialize `.sentris-instance` to `0` and use instance `0`

These scripts intentionally do **not** use plain `TEMPORAL_NAMESPACE` / `TEMPORAL_TASK_QUEUE` by default for the same reason: stale app env files can point a script at the wrong local instance.

Every maintenance script should print its target database and/or Temporal target before mutating data or starting workflows:

```bash
bun --cwd backend scripts/seed-templates.ts --dry-run
# Seed templates dry run: no database writes will be performed.
# Target database: sentris_instance_0 via file:.sentris-instance

bun --cwd backend scripts/seed-templates.ts
# Target database: sentris_instance_0 via file:.sentris-instance

SENTRIS_INSTANCE=2 bun --cwd backend scripts/seed-templates.ts
# Target database: sentris_instance_2 via env:SENTRIS_INSTANCE

TEMPLATE_SEED_DATABASE_URL=postgresql://sentris:sentris@localhost:5433/custom_db bun --cwd backend scripts/seed-templates.ts
# Target database: custom_db via env:TEMPLATE_SEED_DATABASE_URL

bun --cwd backend run migration:push
# Drizzle targets the active instance database

DRIZZLE_DATABASE_URL=postgresql://sentris:sentris@localhost:5433/custom_db bun --cwd backend run migration:push
# Drizzle targets custom_db intentionally

bun --cwd worker scripts/workflow-runner.ts --workflow <workflow-id>
# Target Temporal: namespace=sentris-dev-0, taskQueue=sentris-dev-0 via file:.sentris-instance
```

## E2E Tests (Instance-Aware)

E2E tests use `SENTRIS_INSTANCE`, then legacy `E2E_INSTANCE`, then `.sentris-instance`, initializing it to `0` if missing, to pick the right backend port. The `bun run test:e2e*` scripts call a Node runner, so they work from PowerShell, cmd, Bash, and WSL without requiring `bash` to be installed.

```bash
# Against the active instance
bun run test:e2e

# Against a specific instance
SENTRIS_INSTANCE=2 bun run test:e2e

# Run a scoped E2E folder
bun run test:e2e:core
```

## Troubleshooting

### Port already in use

```bash
# Check which process is using the port
lsof -i :5273   # frontend instance 1
lsof -i :3311   # backend instance 1
```

### Instance is unhealthy but infra is fine

```bash
just instance use 1
just dev logs
just dev stop
just dev
```

### Infra conflicts / stuck containers

```bash
just dev stop    # stops instance 0 + infra
just infra clean
```
