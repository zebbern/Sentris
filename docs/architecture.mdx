## What is Sentris Flow?

Sentris Flow is an open-source, no-code security workflow orchestration platform. Build, execute, and monitor security automation workflows through a visual interface вҖ” focus on security, not infrastructure.

## Security Architecture

### Authentication & Multi-tenancy
- **Clerk Integration** вҖ” Production-ready authentication for hosted environments.
- **Local Auth** вҖ” Default for local setup using `ADMIN_USERNAME` / `ADMIN_PASSWORD`.
- **Organization Isolation** вҖ” All data scoped by `organization_id`.

### Secrets Management
- **AES-256-GCM** encryption at rest.
- **Versioned secrets** with active/inactive tracking.
- **Master key** via `SECRET_STORE_MASTER_KEY` environment variable.

### Container Isolation
- **IsolatedContainerVolume** вҖ” Per-tenant, per-run Docker volumes. See **[Isolated Volumes](/development/isolated-volumes)**.
- **Network isolation** вҖ” Components run with `network: none` or `bridge`.
- **Automatic cleanup** вҖ” Volumes destroyed after execution.
- **No shell execution** вҖ” All Docker commands use `execFile()` to prevent command injection.

### SSRF Protection
- **`validateUrlForSsrf()`** вҖ” The `component-sdk` validates every outbound URL before HTTP requests. Blocks RFC 1918 addresses, link-local (`169.254.x.x`), loopback (`127.x.x.x`), Carrier-Grade NAT (`100.64.x.x`), Docker internal hostnames, and DNS rebinding attacks.

The backend uses [Terminus](https://docs.nestjs.com/recipes/terminus) for health checks. The worker exposes a lightweight HTTP server on port `9100 + N*100` (where N is the instance number).

## Sticky Sessions & MCP Session Registry

MCP (Model Context Protocol) connections are stateful. To ensure a client session always reaches the same backend instance:

1. **Nginx** applies a consistent hash on the `mcp_affinity` cookie for all `/api/v1/mcp/*` routes.
2. **Redis session registry** tracks active sessions under the key pattern `mcp:sessions:{sessionId}`.
3. **Admin endpoint** `GET /api/v1/mcp/sessions` lists all active MCP sessions for debugging and monitoring.

## Observability

### Correlation IDs

Every inbound HTTP request receives a unique `X-Request-Id` (generated by middleware if not already present). This ID:

- Appears in all structured log entries for that request.
- Propagates into Temporal workflow metadata for end-to-end tracing.
- Returns in the HTTP response header for client-side correlation.
