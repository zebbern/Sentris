## What is Sentris Flow?

Sentris Flow is an open-source, no-code security workflow orchestration platform. Build, execute, and monitor security automation workflows through a visual interface — focus on security, not infrastructure.

## Security Architecture

### Authentication & Multi-tenancy
- **Clerk Integration** — Production-ready authentication for hosted environments.
- **Local Auth** — Default for local setup using `ADMIN_USERNAME` / `ADMIN_PASSWORD`.
- **Organization Isolation** — All data scoped by `organization_id`.

### Secrets Management
- **AES-256-GCM** encryption at rest.
- **Versioned secrets** with active/inactive tracking.
- **Master key** via `SECRET_STORE_MASTER_KEY` environment variable.

### Container Isolation
- **IsolatedContainerVolume** — Per-tenant, per-run Docker volumes. See **[Isolated Volumes](/development/isolated-volumes)**.
- **Network isolation** — Components run with `network: none` or `bridge`.
- **Automatic cleanup** — Volumes destroyed after execution.
- **No shell execution** — All Docker commands use `execFile()` to prevent command injection.

### SSRF Protection
- **`validateUrlForSsrf()`** — The `component-sdk` validates every outbound URL before HTTP requests. Blocks RFC 1918 addresses, link-local (`169.254.x.x`), loopback (`127.x.x.x`), Carrier-Grade NAT (`100.64.x.x`), Docker internal hostnames, and DNS rebinding attacks.

## Notification Routing

Notification channels deliver alerts to external services when workflow runs reach a terminal state.

### Event Dispatch

The backend uses **EventEmitter2** (via `@nestjs/event-emitter`) to emit a `run.status.terminal` event whenever a run completes, fails, is cancelled, or times out. The event is fire-and-forget and non-blocking — it does not delay the API response to the client.

### Dispatcher & Adapters

`NotificationDispatcherService` listens for `run.status.terminal` events with `@OnEvent('run.status.terminal', { async: true })`. For each event it:

1. Resolves all active channels whose subscribed events match the run status.
2. Dispatches in parallel via `Promise.allSettled`, delegating to a type-specific adapter (e.g., `SlackNotificationAdapter`).
3. Records each dispatch in the `notification_deliveries` table with status `pending`, then updates to `sent` or `failed`.

New channel types (email, PagerDuty) are added by implementing the `NotificationAdapter` abstract class.

### SSRF Protection for Webhooks

Slack webhook URLs are validated against a domain allowlist (`hooks.slack.com`, `hooks.slack-gov.com`). DNS resolution results are checked to reject internal network IPs, preventing SSRF through DNS rebinding.

### Delivery Tracking

The `notification_deliveries` table stores every dispatch attempt with channel reference, run ID, event type, status (`pending` → `sent` | `failed`), error message, and timestamps. Indexed by `(channel_id, created_at)` and `run_id` for efficient queries.

## Sticky Sessions & MCP Session Registry

MCP (Model Context Protocol) connections are stateful. To ensure a client session always reaches the same backend instance:

1. **Nginx** applies a consistent hash on the `mcp_affinity` cookie for all `/api/v1/mcp/*` routes.
2. **Redis session registry** tracks active sessions under the key pattern `mcp:sessions:{sessionId}`.
3. **Admin endpoint** `GET /api/v1/mcp/sessions` lists all active MCP sessions for debugging and monitoring.

## Observability

### Correlation IDs

Every inbound HTTP request receives a unique `X-Request-Id` (generated by middleware if not already present). This ID:

- Appears in all structured log entries for that request.
- Propagates into Temporal workflow metadata for end-to-end tracing.
- Returns in the HTTP response header for client-side correlation.
